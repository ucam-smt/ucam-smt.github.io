4 \section Basic Translation and Lattice Generation
5 
6 The first demonstration exercise is to generate translations of
7 integer-mapped Russian text using the translation grammar and English
8 n-gram language model provided. HiFST is configured to generate
9 one-best translation hypotheses as well as translation lattices.
10

46 \section basic_latshyps Translation Lattices and 1-Best Hypotheses
47 
48 The baseline configuration file instructs HiFST to write its 1-best
49 translations to the output file `output/exp.baseline/hyps`
50 (see the `target.store=` specification in the config file). The contents of
51 this file should agree with the *Translation 1best* entries in the log file
52 (compare these results to above):
53 
54  > cat output/exp.baseline/hyps
55  1 9121 384 6 2756 7 3 4144 6 1458528 1341 2
56  1 3 1119 6 3 9121 1711 63 355 85 7 369 24 3 13907 17 3 628 5 2
57 
58 The configuration file also directs HiFST to write translation
59 lattices to `output/exp.baseline/LATS/?.fst.gz` (see the
60 `hifst.lattice.store=` specification in the config file). Note the
61 use of the placeholder '`?`' in the argument `.../LATS/?.fst.gz` . The
62 placeholder is replaced by the line number of sentence being
63 translated, e.g. so that `.../LATS/2.fst.gz` is a weighted finite state
64 transducer (WFST) containing
65 translations of the second line in the source text file. Note also
66 the use of the '`.gz`' extension: when this is provided, lattices are
67 written as gzipped files.
68

70 with the top-scoring hypotheses in the hyps and log files :

85 The above operations do the following:
86 
87 -# `zcat` pipes the HiFST output lattice to the \ref OpenFst [Shortest Path](http://openfst.org/twiki/bin/view/FST/ShortestPathDoc) tool which produces an FST containing only the shortest path in the lattice
88 -# the \ref OpenFst [Topological Sort](http://openfst.org/twiki/bin/view/FST/TopSortDoc) operation renumbers the state IDs so that all arcs are links from lower to higher state IDs
89 -# the \ref OpenFst [fstprint](http://openfst.org/twiki/bin/view/FST/FstQuickTour#Printing_Drawing_and_Summarizing) operation reads the English wordmap, for the arc input symbols, and traverses the input fst, writing each arc as it is encountered; TopSort ensures these arcs are written in the correct order
90 -# the awk operation prints only the words on the arcs
91 -# wrapping everything inside echo generates a single string

200 \section basic_scores Scores, Costs, and Semirings
201 
202 HiFST follows the formalism in which rule probabilities are represented as arc weights (see Section 2 of [\ref deGispert2010]).
203 
204 A rule with probability *p* is represented as a negative log probability, i.e.
205 
206  X -> < A , B > / - log(p)
207 
208 with n-gram language model scores encoded similarly,
209 i.e. as costs -log P(w|h)
210 for word *w* with LM history *h*. Costs are accumulated at the
211 path level, so that the shortest path through the output FSA accepts
212 the highest scoring hypothesis under the translation grammar and the
213 language model. Hence the use of \ref OpenFst [ShortestPath](http://openfst.org/twiki/bin/view/FST/ShortestPathDoc) to extract the best scoring
214 hypothesis under the tropical semiring.
215 
216 The \ref OpenFst [Push](http://openfst.org/twiki/bin/view/FST/PushDoc) operation can be used to accumulate weights at the path level within the shortest path fst:
217 
218  > zcat output/exp.baseline/LATS/1.fst.gz | fstshortestpath | fsttopsort | fstpush --push_weights --to_final | fstprint --isymbols=wmaps/wmt13.en.wmap
219  0 1 <s> 1
220  1 2 republican 9121
221  2 3 strategy 384
222  3 4 of 6
223  4 5 resistance 2756
224  5 6 to 7
225  6 7 the 3
226  7 8 renewal 4144
227  8 9 of 6
228  9 10 obamas 1458528
229  10 11 election 1341
230  11 12 </s> 2
231  12 57.4707222,-8.03808594
232 
233 HiFST uses the lexicographic semiring (see [\ref Roark2011]) of two tropical
234 weights to keep track of the translation score and the language model score.
235 
236 The lexicographic semiring is a pair of real-valued weights, and in
237 this application the first component, 54.4707222, contains the sum of the
238 translation grammar scores and the language model. The second component,
239 -8.03808594, is the translation grammar score alone;
240  see Section 5.1 of [\ref Allauzen2014].
241 The lexicographic semiring is such that these scores
242 are computed correctly at the path level:
243 
244 - The cost of the shortest path found by [ShortestPath](http://openfst.cs.nyu.edu/twiki/bin/view/FST/ShortestPathDoc) is that of the best hypothesis under the sum of the translation grammar and language model scores
245 - In the lexicographic semiring, when the path weight is pushed to the final state:
246  - the first weight component is the correct combined translation and language model score
247  - the second weight component is the best translation score over all possible derivations that could have generated this hypothesis
248 
249 One advantage of this representation is that it is easy to remove the language
250 model score prior to [rescoring the lattice with a new language
251 model](#lmrescore) simply by mapping the second component of the
252 lexicographic weight to a plain tropical weight prior to composition
253 with a weighted finite state automaton (WFSA) containing the new language model scores.
254 
255 - N.B. Also see (\ref lmert_veclats_tst)

257 \section basic_toplevelpruning Admissible pruning / top-level pruning
258 
259 HiFST can prune translation lattices prior to saving them to disk. Pruning is
260 done using the \ref OpenFst [Prune](http://openfst.cs.nyu.edu/twiki/bin/view/FST/PruneDoc) operation.
261 Pruning in this case is admissible, since it is
262 performed after translation and language model scores have been
263 completely applied. Low-scoring hypotheses are discarded, but no search
264 errors are introduced by this pruning. Top-level pruning is described in detail in Section 2.2.2, [\ref deGispert2010].
265 
266 Top-level pruning is controlled by the `--hifst.prune` option. In the
267 previous examples, `--hifst.prune` was set to 9. If we use the default
268 (3.40282347e+38), then the output lattice size becomes very large. For
269 example, compare lattices in `exp1/` generated with `prune=9` vs.
270 unpruned lattices in `exp2/`:
271

325 \section multithread Multithreading
326 
327 (* note that the timing results here are illustrative only.)
328 
329 HiFST uses
330 [Boost.Thread](http://www.boost.org/doc/libs/1_38_0/doc/html/thread.html)
331 to enable multithreading. This is disabled by default, but can enabled using
332 the flag `--nthreads=N` . If set, each source language sentence is
333 translated simultaneously on its own thread (trimmed to the number of
334 CPUs available). The translation grammar and language model are kept in
335 shared memory.
336 
337 To see the effects of multithreading on speed and memory use,
338 the baseline configuration is run over the first twenty sentences without multithreading:
339 
340  > time hifst.O2 --config=configs/CF.baseline --range=1:20
341 
342 Processing time is 140 seconds and maximum memory use is about 0.3GB.
343 In the same decoder configuration but with 2 threads
344 
345  > time hifst.O2 --config=configs/CF.baseline --range=1:20 --nthreads=2
346 
347 processing time is reduced to 90 seconds with maximum memory use of about 0.5GB.
348 
349 In these examples, both the LM and translation grammar are relatively
350 small, and so there is not a great deal of gain from keeping them in
351 shared memory. But in larger tasks, multithreading can be a significant advantage.

353 \section server Client-Server Mode (Experimental)
354 
355 HiFST can run in server mode.
356 
357  > hifst.O2 --config=configs/CF.baseline.server &> log/log.server &
358  > pid=$! # catch the server pid
359 
360 Note that in this particular configuration, both source and target wordmaps are loaded.
361 Hifst can read tokenized Russian text and produce tokenized English translations
362 (see options `--prepro.wordmap.load` and `--postpro.wordmap.load`).
363 Also, to ensure that CYK parser never fails, out of vocabulary (OOV) words must be detected (`--ssgrammar.addoovs.enable`) and sentence markers (`<s>`,`</s>`)
364 have to be added on the fly, as the shallow grammar relies on them (i.e. `S 1 1`).
365 
366 With the `hifst-client.O2` binary, we can read Russian tokenized text (`RU/RU.tune`) and submit translation requests to the server.
367 The output is stored in a file specified by the client tool (`--target.store`).
368 
369  > sleep 60 # make sure to wait for the server to finish loading, otherwise clients will fail
370  > hifst-client.O2 --config=configs/CF.baseline.client --range=200:5:300 --target.store=output/exp.clientserver/translation1.txt &> log/log.client1 &
371  # Connect to localhost, port=1205 and translate a bunch of sentences. Lets do this in background, just for fun
372  # Note that the localhost setting is in the config file; this can point to another machine, of course
373  > pid2=$!
374 
375  > hifst-client.O2 --config=configs/CF.baseline.client --range=1:50,100,1300 --target.store=output/exp.clientserver/translation2.txt &> log/log.client2 &
