1 \subsection lpruning Local pruning / pruning in search
2 
3 Local pruning controls processing speed and memory use during translation.
4 Only enough details are reviewed here to describe how HiFST performs pruning in search;
5 for a detailed discussion of local pruning and pruning in search, see Section 2.2.2 of [\ref deGispert2010].
6 
7 Given a translation grammar and a source language sentence, HiFST first
8 constructs a Recursive Transition Network (RTN) representing
9 the translation hypotheses [\ref Iglesias2009, \ref Iglesias2011].
10 This is done as part of a modified CYK algorithm used to parse the
11 source sentence under the translation grammar.
12 The RTN is then *expanded* to an equivalent WFSA via the \ref OpenFst [Replace](http://openfst.cs.nyu.edu/twiki/bin/view/FST/ReplaceDoc)
13 operation. This WFSA contains the translation hypotheses along with their scores under the translation grammar.
14 We refer to this as the `top-level' WFSA, because it is associated with the top-most cell in the CYK grid.
15 This top-level WFSA can be pruned after composition with the language
16 model, as described in the discussion of \ref basic_toplevelpruning.
17 We refer to this as *exact search* or *exact translation*.
18 In exact translation, no translation hypotheses are discarded prior to applying the complete translation and language model scores.
19 
20 Exact translation can be done under some combinations of translation grammars, language models, and language pairs.
21 In particular, the \ref tgrammars_shallow were designed for exact search.
22 However attempting exact translation under many translation grammars would cause
23 either the [Replace](http://openfst.cs.nyu.edu/twiki/bin/view/FST/ReplaceDoc)
24 operation or the subsequent language model composition to
25 become computationally intractable.
26 We therefore have developed a pruning strategy that prunes the RTN during its construction.
27 
28 The RTN created by HiFST can be described as follows:
29  - \f$X\f$ is the set of non-terminals in the translation grammar, with \f$S\f$ as the root
30  - \f$\Sigma\f$ is the target language vocabulary, i.e. the terminals in the target language
31  - \f$I\f$ is the length of the source sentence \f$s\f$, i.e. \f$s = s_0...s_{I - 1}\f$
32  - A new set of non-terminals is defined as \f$N = \{ (x,i,j) : x \in X , 0 <= i <= j < I \}\f$
33  - Note that \f$(S,0,I-1) \in N\f$
34  - \f$(T_u)_{u \in N}\f$, is a family of WFSAs with input alphabet \f$\Sigma \cup N\f$
35  - Each \f$T_u\f$ with \f$u = (x, i, j)\f$, is a WFSA that describes all applications of translation rules with left-hand side non-terminal \f$x\f$ that
36 span the substring \f$s_i ... s_j\f$
37  - \f$T_u\f$ is associated with the CYK grid cell associated with source space \f$[i,j]\f$ and headed by non-terminal \f$x\f$
38  - The top-level RTN is defined as \f$R_{(S,0,I-1)} = (N, \Sigma, (T_u)_{u \in N}, (S,0,I-1))\f$.
39  - The root symbol of this RTN is \f$(S,0,I-1)\f$.
40  - The WFSA \f$T_{(S,0,I-1)}\f$ represents all applications of translation rules that span the entire sentence and are rooted with non-terminal \f$S\f$.
41 
42 Exact translation is achieved if every \f$T_u\f$ is complete (i.e. if no pruning is done) prior to the \ref OpenFst [Replace](http://openfst.org/twiki/bin/view/FST/ReplaceDoc)
43 operation
44 on the RTN \f$R_{(S,0,I-1)}\f$. This produces a WFSA that contains all translations
45 that can be produced under the translation grammar.
46 
47 The RTN pruning strategy relies on noting
48 that each of the WFSAs \f$T_{u'}\f$, \f$u' = (x', i', j')\f$, also defines an RTN \f$R_{u'}\f$, as follows:
49  - Define a subset of non-terminals \f$N' = \{ (x,i,j) : x \in X , i' <= i <=j < j' \}\f$ , i.e. \f$N' \subset N\f$
50  - \f$R_{u'} = (N', (T_u)_{u \in N'}, (x', i', j') )\f$
51  - The root symbol of this RTN is \f$(x', i', j')\f$
52 
53 The [Replace](http://openfst.cs.nyu.edu/twiki/bin/view/FST/ReplaceDoc)
54 operation can be applied to the RTNs \f$R_{u'}\f$ to produce an WFSA
55 containing all translations of the source string \f$S_{i'} ... S_{j'}\f$ using
56 derivations rooted in the non-terminal \f$x'\f$. This WFSA can be pruned and used in place of the original \f$T_{u'}\f$.
57 
58 Because of the possibility of search errors we refer to this as 'local pruning' or inadmissible pruning.
59 There is the possibility that pruning any of the \f$T_u\f$ may possibly cause some good translations to be discarded.
60 For this reason it is important to tune the pruning strategy for the translation grammar and language model.
61 Once pruning has been set, the benefits are
62  - faster creation of the top-level WFSA via the [Replace](http://openfst.cs.nyu.edu/twiki/bin/view/FST/ReplaceDoc) operation
63  - faster composition of the translation WFSA with the language model
64  - less memory used in RTN construction and language model composition
65 
66 Local pruning should be done under the combined
67 translation and the language model scores, rather than under the translation grammar scores alone
68 alone. However, the LM used in local pruning can be relatively weak. For
69 example, if the main language model used in translation is a 4-gram,
70 perhaps a 3-gram or even a bigram language model could be used in
71 local pruning. Using a smaller language model will make pruning faster, as
72 will an efficient scheme to remove the scores of the language models used in pruning.
73 The lexicographic semiring, see \ref basic_scores, makes this last operation easy.
74 
75 \subsubsection local_prune Local Pruning Algorithm
76 
77 HiFST monitors the size of the \f$T_u\f$ during translation. Any of these automata that exceed specified thresholds
78 are converted to WFSAs and pruned. Subsequent expansion of the RTN \f$R_{(S,0,I-1)}\f$ is then done with respect to the pruned
79 versions of \f$T_u\f$.
80 
81 
82 Local pruning is controlled via the following HiFST parameters:
83 
84  hifst.localprune.enable=yes # must be set to activate local pruning
85  hifst.localprune.conditions=NT_1,span_1,size_1,threshold_1,...,NT_N,span_N,size_N,threshold_N
86  hifst.localprune.lm.load=lm_1,...lm_K
87  hifst.localprune.lm.featureweights=scale_1,...,scale_K
88  hifst.localprune.lm.wps=wp_1,...,wp_K
89 
90 In the above, an arbitrary number N of tuples (`NT_n`, `span_n`, `size_n`, `threshold_n`) can be provided;
91 similarly, an arbitrary number K of language model parameters (`lm_k`, `scale_k`, `wp_k`) can also be used in pruning.
92 
93 Pruning is applied during construction of the RTN, as follows:
94 
95  - If any \f$T_u\f$ satisfies the following conditions for any parameter set (`NT_n`, `span_n`, `size_n`, `threshold_n`), n=1,...,N
96  - NT_n = X
97  - span_n <= j-i
98  - size_n <= number of states of \f$T_u\f$, computed via \ref OpenFst `NumStates()`
99  - then \f$T_u\f$ is pruned as follows:
100  - OpenFst [Replace](http://openfst.cs.nyu.edu/twiki/bin/view/FST/ReplaceDoc) converts \f$R_u\f$ to a WFSA
101  - [RmEpsilon](http://www.openfst.org/twiki/bin/view/FST/RmEpsilonDoc),[Deteminize](http://www.openfst.org/twiki/bin/view/FST/DeterminizeDoc), and [Minimize](http://www.openfst.org/twiki/bin/view/FST/MinimizeDoc) generate a compacted WFSA
102  - [Composition](http://www.openfst.org/twiki/bin/view/FST/ComposeDoc) with K language model(s) WFSAs
103  - The parameters (`lm_k`, `scale_k`, `wp_k`) specify the language models, language model scale factors, and word penalties to be applied
104  - OpenFst [Prune](http://www.openfst.org/twiki/bin/view/FST/PruneDoc) is applied with threshold `threshold_n`
105  - Language model scores are removed by copying component weights in the lexicographic semiring, see \ref basic_scores
106  - [RmEpsilon](http://www.openfst.org/twiki/bin/view/FST/RmEpsilonDoc),[Deteminize](http://www.openfst.org/twiki/bin/view/FST/DeterminizeDoc), and [Minimize](http://www.openfst.org/twiki/bin/view/FST/MinimizeDoc), yielding a pruned WFSA \f$T_u\f$ with only translation scores and target language symbols
107  - The pruned version of \f$T_u\f$ is then used in place of the original version in the RTN

189 \section chopping Source Sentence Chopping
190 
191 Long source sentences make the translation process slow and expensive
192 in memory consumption; see [\ref Allauzen2014] for a discussion of how
193 source sentence length affects computational complexity and memory use
194 by HiFST and HiPDT. There are various strategies for controlling
195 translation complexity; pruning has been discussed (\ref lpruning), and it is also
196 possible to set the maximum span and gap spans allowed in translation
197 so as to control computational complexity. However translation quality
198 can be affected if pruning is too heavy or if span constraints are set
199 too aggressively.
200 
201 An alternative approach is to 'chop' long sentences into shorter
202 segements which can then be translated separately. If the sentence
203 chopping is done carefully, the impact on the translation quality can
204 be minimized. The benefits to chopping are faster translation that
205 consumes less memory. The potential drawbacks are twofold: chopping
206 can prevent the search procedure from finding good hypothesis under
207 the grammar, and care must be taken to correctly apply the target
208 language model at the sentence level.
209 
210 \subsection chopping_gb Grammar-based Sentence Chopping
211 
212 Chopping can be done by inserting the special 'chop' symbol
213 '0' in the source sentence, and then translating with a modified
214 grammar. The chopping grammar is constructed so that translation
215 rules are not applied across the chopping points, thus limiting the
216 space of translation that are generated. Conceptually, translation proceeds as:
217  -# the translation grammar is applied separately to source sentence segments demarcated by chop symbols
218  -# local pruning can be applied to the translations of these segments
219  -# the resulting WFSAs containing translations of the segments are concatenated under the chopping grammar, possibly with local pruning
220  -# the language model is applied
221  -# top-level, admissible pruning is done under the translation and languaage model scores
222 
223 In this way the FSTs produced by translating the segments are
224 concatenated prior to application of the target language model; in
225 this way the language model context is not broken by the source
226 sentence chopping.
227 
228 As an example, a grammar modified for chopping contains the following rules (without
229 weights):
230 
231  R 1 1
232  R R_D_X R_D_X
233  R R_X R_X
234 
235  T 0 0
236  T T_D_X T_D_X
237  T T_X T_X
238 
239  S S_U S_U
240  S Q Q
241  Q R R
242  U T T
243 
244 The rules in the first block above are similar to those used in the
245 usual Hiero grammar, with the original '`S`' changed to '`R`'. These rules
246 are responsible for concatenating the partial translations of the source
247 sentence, starting from the sentence-start symbol
248 '1', up to but not including the first instance of the chopping symbol
249 '0'.
250 
251 Each subsequent sequence of source words starting with symbol '0', is
252 handled in a similar way by the second block of rules above. Note that
253 the only rule that can be applied to the input symbol '0' is '`T 0 0`', making the
254 translation of each chopped segment independent. This makes use of
255 the OpenFST convention of mapping 0 to epsilon: the 0's in the input
256 are parsed as regular symbols by HiFST, while 0's on the output side
257 are mapped to epsilons and ignored in composition with the language
258 model.
259 
260 The third block of rules above will join together the results obtained
261 for each chopped segment. As with the glue rule '`S`' in the usual Hiero
262 grammar, it is necessary to allow this new set of rules to be applied
263 to any span. This is done by setting
264 
265  cykparser.ntexceptionsmaxspan=S,Q,R,T,U
266 
267 The additional mapping provided by the last two rules controls
268 the pruning applied to the top CYK cell relative to each
269 chopped segment:
270 
271  hifst.localprune.conditions=Q,1,100,12,U,1,100,12,X,5,10000,9,V,3,20000,9
272 
273 In the above example tighter parameters are chosen for '`Q`' and '`U`' to
274 force pruning. In this way the final lattice obtained by
275 concatenation is prevented from growing too large. However a wider
276 beam (12) with respect to the other cell types is used, to avoid
277 discarding too many potentially useful hypotheses.
278 
279 It is possible to specify explicitly that FSTs generated for rules
280 with LHS '`X`' or '`V`' can be kept as pointers rather then expanded in
281 the FST (RTN) that is built for a higher CYK cell. This is achieved
282 setting
283 
284  hifst.replacefstbyarc=X,V
285  hifst.replacefstbyarc.exceptions=S,R,T
286 
287 The second line above prevents substitution for rules with LHS
288 '`S`', '`R`' and '`T`'. It is better to have a fully expanded FST for these
289 rules for more effective optimisation (Determinization and Minimisation).

293 The usual Hiero grammar can be converted for chopping, as follows; note that no-cost, 0 valued, weights are added to rules :
294
