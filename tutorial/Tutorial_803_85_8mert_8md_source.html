89 \subsection mert_nblist_derivations 2. Guided Translation / Forced Alignment
90 
91 - Input:
92  - `RU/RU.tune.idx` -- tuning set source language sentences
93  - `G/rules.shallow.all.gz` -- translation grammar
94  - `M/interp.4g.arpa.newstest2012.tune.corenlp.ru.idx.union.mmap` -- target language model
95  - `output/exp.mert/LATS/?.fst.gz` -- word lattices (WFSAs), determinized and minimized (from \ref mert_hyps)
96 - Output:
97  - `output/exp.mert/nbest/ALILATS/?.fst.gz` -- transducers mapping derivations to translations (e.g. Fig. 7, [\ref deGispert2010])
98 
99 Alignment is the task of finding the derivations (sequences of rules)
100 that can produce a given translation. HiFST performs alignment via
101 constrained translation (see Section 2.3, [\ref deGispert2010] for a
102 detailed description). This command runs HiFST in alignment mode:
103 
104  > hifst.O2 --config=configs/CF.mert.alilats.nbest --range=1:$M &> log/log.mert.alilats.nbest
105 
106 In alignment mode, HiFST constructs *substring acceptors* (see Fig. 8,
107 [\ref deGispert2010]). These are constructed for each sentence as follows:
108 
109 - the lattice from Step 1 is loaded by HiFST
110 - an N-Best list in the form of a WFSA is extracted (using `fstshortestpath`) under the translation and language model score from Step 1
111 - weights are removed from N-Best WFSA
112 - the WFSA is transformed to a substring acceptor
113 
114 The translation grammar is applied in the usual way, but the
115 translations are intersected with the substring acceptors so that only
116 translations in the N-Best lists are retained. This generates every
117 possible derivation of each N-Best list entry.
118 
119 This behaviour is specified by the following config file parameters:
120 
121  [referencefilter]
122  load=output/exp.mert/LATS/?.fst.gz
123  # perform alignment against these reference lattices containing initial hypotheses
124  prunereferenceshortestpath=100
125  # on loading the reference lattices, transform them to n-best lists prior to alignment.
126  # uses fstshortestpath
127 
128 Note that application of the substring acceptors is very efficient;
129 and this alignment step should be much faster than the translation
130 operation of Step 1.
131 The alignment lattices (referred to as ALILATS) map rule sequences
132 (derivations) to translation hypotheses. Weights remain in
133 lexicographic semiring form.
134 
135  > zcat output/exp.mert/nbest/ALILATS/1.fst.gz | fstinfo | head -n 2
136  fst type vector
137  arc type tropical_LT_tropical
138 
139 Individual rules are identified by their line number in the translation grammar file. A rule map can be created as
140 
141  > zcat G/rules.shallow.all.gz | awk 'BEGIN{print "0\t0"}{printf "%s-><%s,%s>\t%d\n", $1, $2, $3, NR}' > G/rules.shallow.all.map

143

154 The order of the rules in these rule sequences correspond to HiFST's
155 bottom-up (left-to-right) CYK grid structure. Rule IDs are added as
156 input symbols to the component WFSTs in the RTN following the
157 translation rule (with its non-terminals). This leads to the
158 bottom-up ordering after Replacement.

161 \subsection mert_alilats 3. Hypotheses with Unweighted Feature Vectors
162 
163 - Input:
164  - `G/rules.shallow.vecfea.all.gz` -- translation grammar, rules with (unweighted) feature vectors
165  - `output/exp.mert/nbest/ALILATS/?.fst.gz` -- transducers mapping derivations to translations, for n-best entries (from \ref mert_nblist_derivations)
166  - language model and translation grammar feature weights (see `configs/CF.mert.vecfea.nbest`)
167 - Output:
168  - `output/exp.mert/lats/VECFEA/?.nbest.gz` -- N-best hypotheses
169  - `output/exp.mert/lats/VECFEA/?.vecfea.gz` -- N-best unweighted features
170 
171 The alilats2splats tool transforms ALILATS alignment lattices
172 (transducers) to sparse vector weight lattices; see Section 2.3.1, [\ref deGispert2010] for a detailed explanation.
173 
174  > alilats2splats.O2 --config=configs/CF.mert.vecfea.nbest --range=1:$M &> log/log.mert.nbest
175 
176 The output is written to two sets of files:
177 
178 N-best lists:
179 
180  > head -n 2 output/exp.mert/nbest/VECFEA/1.nbest
181  <s> parliament does not support the amendment , which gives you the freedom of tymoshenko </s> 43.0904
182  <s> the parliament does not support the amendment , which gives you the freedom of tymoshenko </s> 43.1757
183 
184 Vecfea files:
185 
186  > head -n 2 output/exp.mert/nbest/VECFEA/1.vecfea
187  62.5442 10.8672 8.3936 -16.0000 -8.0000 -5.0000 0.0000 -1.0000 0.0000 -7.0000 16.3076 40.5293
188  63.1159 12.8613 8.7959 -17.0000 -8.0000 -5.0000 0.0000 -1.0000 0.0000 -7.0000 17.0010 43.9482
189 
190 - Line `n` in the nbest list is the `n-th' translation hypotheses, as ranked under the combined translation and language model scores.
191 - Line `n` in the vecfea file is a vector obtained by summing the unweighted feature vectors of each rule in the best derivation of the `n-th` hypothesis
192 
193 The alilats2splats tool works as follows:
194 
195 - The translation grammar with (unweighted) feature vectors is loaded
196 - a Rule Flower acceptor, R, is created. This is an acceptor for rule sequences that applies vector weights (specifically, the feature vector for each rule). See Fig. 9 of [\ref deGispert2010] for an example and an explanation.
197 - For each source sentence, the ALILATS derivation-to-translation transducer from Step 2 is loaded, and its weights are removed. Call this T_u
198 - The unweighted derivations-to-translation trandsducer T_u is composed with the Rule Flower acceptor R under the tropical sparse tuple weight semiring with the same feature vectors as are used to generate the translation.
199 - The feature vector for the best scoring derivation for every translation is found as Determinise(Project_output(R o T_u) )
200 - Language model scores M_1, ..., M_m are applied (again in the tropical sparse tuple weight semiring, so that each score ends up in a separate element in the vector) as Determinise(Project_output(R o T_u) ) o M_1 o ... o M_m
201 
202 Writing of N-Best lists and features is
203 controlled by the `sparseweightvectorlattice` options `storenbestfile`
204 and `storefeaturefile`:
205 
206  [sparseweightvectorlattice]
207  loadalilats=output/exp.mert/nbest/ALILATS/?.fst.gz
208  storenbestfile=output/exp.mert/nbest/VECFEA/?.nbest
209  storefeaturefile=output/exp.mert/nbest/VECFEA/?.vecfea
210  wordmap=wmaps/wmt13.en.all.wmap
211 
212 With the wordmap specified, the output of alilats2splats is in readable form in the
213 target language. Note that the sentence boundary symbols and the combined
214 translation and language model score appear in the nbest file.
215 The N-best lists have the format
216 - wordindex1 wordindex2 ... translation_score
217 
218 The relationship of feature vectors and scores at the hypothesis level is as follows:
219 - Suppose there are m language models, with weights s_1 ...,s_m .
220  - These weights are specified by the HiFST parameters `lm.featureweights=s_1,s_2,..,s_m`
221 - Suppose there are n dimensional feature vectors for each rule,
222  - The weights to be applied are specified by the HiFST parameters `grammar.featureweights=w_1,..,w_n`
223 - A feature weight vector is formed as P = [s_1 ... s_m w_1 ... w_n]
224 - A translation hypothesis e has a feature vector F(e) = [lm_1(e) ... lm_m(e) f_1(e) ... f_n(e)]
225  - lm_i(e): the i-th language model score for e
226  - f_j(e): j-th grammar feature (see Section 3.2.1, [\ref deGispert2010])
227 - The score of translation hypothesis e can be found as S(e) = F(e) . P (dot product)
228 
229 Each line k in the feature file has the format
230 - lm_1(e_k) ... lm_m(e_k) f_1(e_k) ... f_n(e_k)
231 
232 which are the unweighted feature values for the k-th hypothesis, e.g.
233 
234  > head -n 2 output/exp.mert/nbest/VECFEA/1.vecfea
235  62.5442 10.8672 8.3936 -16.0000 -8.0000 -5.0000 0.0000 -1.0000 0.0000 -7.0000 16.3076 40.5293
236  63.1159 12.8613 8.7959 -17.0000 -8.0000 -5.0000 0.0000 -1.0000 0.0000 -7.0000 17.0010 43.9482
237 
238 and the translation_score in line k is F(e_k) . P
