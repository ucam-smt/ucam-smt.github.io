3 \section gext Using Grammars From Open Source SMT Systems
4 
5 The current release of HiFST does not include our rule extraction system.
6 For this tutorial, we demonstrate how to convert grammars obtained from
7 other systems into the HiFST format.
8 All files following the above format can be used in HiFST. Here we briefly introduce ways for adapting SCFGs generated by several existing SMT systems to the HiFST format.
9 
10 \subsection gext_moses Moses
11 
12 <http://www.statmt.org/moses/>
13 
14 The Moses package provides utilities for both phrase extraction and hierarchical rule extraction. Suppose that we have a pair of sentences and the word-level alignment between them:
15 
16  SOURCE = 61 10717 3267 486 3 6501 3 53 1230 44 4 902 7631 2213 16 2265 319 5
17  TARGET = 71 29 5558 5494 831 3 1331 2242 7 1250 5 1003 13 439 8 5663 601 4 712 4 7 86 6
18  ALIGNMENT = 0-0 0-1 1-2 2-3 8-3 1-4 10-5 11-6 12-7 13-7 14-8 16-9 15-11 16-13 1-15 3-16 4-17 5-18 6-19 7-21 17-22
19 
20 We can obtain a phrase translation table using Moses, like this,
21 
22 
23  ...
24  44 4 902 7631 2213 ||| 3 1331 2242 ||| 0.5 0.25 1 1 2.718 ||| ||| 2 1
25  44 4 902 ||| 3 1331 ||| 0.5 1 1 1 2.718 ||| ||| 2 1
26  44 4 ||| 3 ||| 0.5 1 1 1 2.718 ||| ||| 2 1
27  ...
28 
29 
30 It is trivial to transform the above phrase pairs into the HiFST
31 rules: we only need to introduce a non-terminal symbol into the LHSs
32 of the rules. Here we take X as the only one non-terminal symbol. Then
33 we concatenate the sequence of words with an underscore `_` on both language sides,
34 e.g., `44 4 902 7631 2213` => `44_4_902_7631_2213`. Then the weights
35 can be used by performing a -log() computation to fit for the tropical
36 semiring used in HiFST.
37 
38 Note that HiFST prefers the use of
39 target-word count as a feature in the SCFG to eliminate the bias
40 towards short sentences caused by the use of n-gram language model. So
41 here we add the negative target-word count into the weight vector as a new
42 feature, e.g, for the first phrase pair, the first weight should be -3
43 (i.e., minus target-word number). By using the above methods, we get
44 the rules in the HiFST format, as follows
45 
46  ...
47  X 44_4_902_7631_2213 3_1331_2242 -3 0.693 1.386 0 0 -1
48  X 44_4_902 3_1331 -2 -0.693 0 0 0 -1
49  X 44_4 3 -1 -0.693 0 0 0 -1
50  ...
51 
52 In addition to the phrasal translations, Moses provides a tool for hierarchical phrasal rule extraction. For the above example, the Moses hierarchical rule extractor can generate a rule file:
53 
54 
55  ...
56  44 [X][X] 16 [X] ||| [X][X] 7 [X] ||| 0.084 0.5 1 1 2.718 ||| 1-0 ||| 1.074 0.090
57  44 [X][X] 16 [X][X] [X] ||| [X][X] 7 [X][X] 8 [X] ||| 0.125 0.5 0.275 0.25 2.718 ||| 1-0 3-2 ||| 0.307 0.139
58  44 [X][X] 16 [X][X] [X] ||| [X][X] 7 [X][X] [X] ||| 0.128 0.5 0.724 1 2.718 ||| 1-0 3-2 ||| 0.786 0.139
59  ...
60 
61 On each language side of a rule, `[X]` represents the non-terminal
62 symbol X of the left hand side (Note: both languages share the same
63 LHS). `[X][X]` represents a non-terminal with a non-terminal symbol
64 `X`. If more than one non-terminals are involved, we need the alignment
65 information to determine the relative order of non-terminals. This can
66 be found in the 4-th field of each rule. E.g., for the second rule in
67 the above example, we can access the alignment information "1-0 3-2"
68 and know that it is a monotonic translation rule. Hence all these
69 hierarchical phrasal rules in Moses can be transformed into the rules
70 used in HiFST, like so
71 
72  ...
73  X 44_X_16 X_7 -1 2.476 0.693 0 0 -1
74  X 44_X1_16_X2 X1_7_X2_8 -2 2.079 0.693 1.290 1.386 -1
75  X 44_X1_16_X2 X1_7_X2 -1 2.055 0.693 0.322 0 -1
76  ...
77 
78 
79 As HiFST uses a single grammar file as input, we can concatenate the phrase translation file and the hierarchical rule file to form the final SCFG file.
80 
81 \subsection gext_joshua Joshua and cdec
82 
83 <http://joshua-decoder.org/> and <http://cdec-decoder.org/index.php?title=Main_Page>
84 
85 Joshua and cdec support a different definition of SCFG files, where each rule follows the format of
86 
87  [LHS] ||| RHS_SOURCE ||| RHS_TARGET ||| FEATURES(WEIGHTS)
88 
89 For the above example, one can obtain the following rule file in the Joshua format.
90 
91 \code
92  ...
93  [X] ||| 44 4 902 7631 2213 ||| 3 1331 2242 ||| -0.693 -1.386 0 0 1
94  [X] ||| 44 4 902 ||| 3 1331 ||| 0.693 0 0 0 1
95  [X] ||| 44 4 ||| 3 ||| 0.693 0 0 0 1
96  ...
97  [X] ||| 44 [X,1] 16 ||| [X,1] 7 ||| -2.476 -0.693 0 0 1
98  [X] ||| 44 [X,1] 16 [X,2] ||| [X,1] 7 [X,2] 8 ||| -2.079 -0.693 -1.290 -1.386 1
99  [X] ||| 44 [X,1] 16 [X,2] ||| [X,1] 7 [X,2] ||| -2.055 -0.693 -0.322 0 1
100  ...
101 \endcode
102 
103 Here `[X]` represents the non-terminal symbol on the left-hand side of a rule. `[X,n]` represents a non-terminal on the right hand side, where n is an index to determine the non-terminal order. All features(weights) are in log-scale. So we use minus operation to fit them into the weight definition preferred by HiFST. Also, the (minus) target-word number is added as an additional feature. For this Joshua version of the SCFG file, we can transform it into the HiFST-style file:
104 
105 
106 \code
107  ...
108  X 44_4_902_7631_2213 3_1331_2242 -3 0.693 1.386 0 0 -1
109  X 44_4_902 3_1331 -2 -0.693 0 0 0 -1
110  X 44_4 3 -1 -0.693 0 0 0 -1
111  ...
112  X 44_X_16 X_7 -1 2.476 0.693 0 0 -1
113  X 44_X1_16_X2 X1_7_X2_8 -2 2.079 0.693 1.290 1.386 -1
114  X 44_X1_16_X2 X1_7_X2 -1 2.055 0.693 0.322 0 -1
115  ...
116 \endcode

119 \subsection gext_niutrans NiuTrans
120 
121 <http://www.nlplab.com/NiuPlan/NiuTrans.html>
122 
123 NiuTrans uses another format for SCFG files, like
124 
125 \code
126  RHS_SOURCE ||| RHS_TARGT ||| LHS ||| FEATURE (WEIGHTS)
127 \endcode
128 
129 Based on the NiuTrans format, the above sample grammar can be written, as follows:
130 
131 
132 \code
133  ...
134  44 4 902 7631 2213 ||| 3 1331 2242 ||| X ||| -0.693 -1.386 0 0 1
135  44 4 902 ||| 3 1331 ||| X ||| 0.693 0 0 0 1
136  44 4 ||| 3 ||| X ||| 0.693 0 0 0 1
137  ...
138  44 #X 16 ||| #1 7 ||| X ||| -2.476 -0.693 0 0 1
139  44 #X 16 #X ||| #1 7 #2 8 ||| X ||| -2.079 -0.693 -1.290 -1.386 1
140  44 #X 16 #X ||| #1 7 #2 ||| X ||| -2.055 -0.693 -0.322 0 1
141  ...
142 \endcode
143 
144 On the source-language side, each non-terminal is marked with '#', followed by its symbol. E.g., '\#X' is a non-terminal with the symbol X. On the target-language side, each non-terminal is also marked with '#' but uses an integer to indicate the index of the corresponding non-terminal in the source-language side. E.g., in the last rule of the above example, #2 is a non-terminal on the target-language side. It is aligned with the second non-terminal on the source-language side and shares the symbol X with its counterpart.
145 
146 It is not difficult to transform a NiuTrans-style grammar file into a HiFST-style grammar file. For this example, we have (the same result again)
147 
148 \code
149  ...
150  X 44_4_902_7631_2213 3_1331_2242 -3 0.693 1.386 0 0 -1
151  X 44_4_902 3_1331 -2 -0.693 0 0 0 -1
152  X 44_4 3 -1 -0.693 0 0 0 -1
153  ...
154  X 44_X_16 X_7 -1 2.476 0.693 0 0 -1
155  X 44_X1_16_X2 X1_7_X2_8 -2 2.079 0.693 1.290 1.386 -1
156  X 44_X1_16_X2 X1_7_X2 -1 2.055 0.693 0.322 0 -1
157  ...
158 \endcode

160 \section grammarpruning Translation Grammar Pruning
161 
162 Once the grammar is obtained, one may consider to prune it for a more compact model as well as a more efficient decoding process. Note that grammar pruning is widely used in SMT systems. This is especially important for some large-scale translation tasks, e.g., training a Chinese-English SMT system on millions of sentence pairs. Without appropriate pruning, the translation speed might be unacceptable in some cases. Here we present a few tips on grammar pruning for better use of HiFST.
163 
164 1. For the source-language of each phrase/rule, we can keep the top-n target-sides in terms of (source-to-target) translation probability. This method is adopted in most open source systems by default. Generally setting n = 10~30 is enough for most tasks.
165 
166 2. We can prune away low frequency phrases/rules. For example, there is generally much noise in hierarchical phrase translation rules that occur once in the training data. So it is relatively "safe" to discard this type of rule.
167 
168 3. Also, we can only keep hierarchical phrase rules that are extracted on a good-quality portion of the parallel corpus, that is, we still obtain phrasal translations from the whole corpus but extract hierarchical phrase rules from a smaller set of sentences.
169 
170 4. Another method is to filter rules according to rule patterns. For example, both rule patterns "X -> X w, X w" and "X -> w X, w X" explain monotonic translation phenomena (where X is a variable and w is a sequence of words). In this case, we can throw away one of this patterns because they are actually doing the same thing.
171 
172 5. In addition to rule filtering, we can also write the grammar to another form for more efficient decoding. For example, we can transform an SCFG to a sallow grammar which avoids unlimited nested structures and makes the decoding much faster. Please see next section for more discussion on shallow grammars.
173 
174 The reader can refer to \ref Iglesias2009b for more methods for grammar pruning.
